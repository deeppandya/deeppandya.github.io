⚠️ The shared link doesn't open the Android app.

All three Share buttons emit the same URL — https://whop.com/live/{feedId} (web share-livestream-modal.tsx, iOS LivestreamViewModel.swift, Android LivestreamViewModel.kt). Web resolves it (/live/[feedId] → experience) and iOS resolves it (parses lfeed_, fetches the shared experience), but Android has no /live/ parser case or App-Links entry, so a tapped /live/… link opens the browser. Android only deep-links the hub form /hub/{company}/livestreaming-{experienceId}/app. See row 3.

Android's livestream parser requires the literal hub/ segment, so this falls to the generic experience parser rather than the livestream launch path.

Android path shape: /hub/{companyId}/livestreaming-{experienceId}/app (the hub/ segment + livestreaming- prefix on the 3rd segment are what the parser keys on). A push notification can override the target by sending clientData.livestream_feed_id; otherwise the parsed experience id is used as the fallback. iOS & web additionally resolve the bare /live/{feedId} share link.

Android testing (develop build)

Build: com.whop.whopapp.develop  ·  signed in as @deeppandya91.

Browser tap. The hub route matches the manifest App-Links pattern /.*/.*/app, so a tapped /hub/…/livestreaming-…/app link opens the app. On the develop build, verification isn't automatic — enable it once in Settings → Apps → Whop (develop) → Open by default → Open supported links → add whop.com. The /live/{feedId} share link is not registered, so it always opens the browser.

adb (always works, bypasses link verification). Pushes the URL as the deeplink extra. Only the hub route resolves to a livestream launch; /live/… is parsed as null and no activity is started:

adb shell "am start -n com.whop.whopapp.develop/com.whop.whopapp.MainActivity \
  --es deeplink '<URL>'"

Android-specific behavior notes

  • Livestream deep links don't navigate via a route — the handler fires Event.StartLiveStreamEvent to launch LivestreamActivity.
  • Target resolution: clientData.livestream_feed_id (push payload) wins; otherwise the parsed experienceId is the fallback.
  • Deep links delivered while the app is backgrounded/cold are replayed once the UI is ready.
  • The /live/{feedId} share link is not handled — to deep-link a specific feed today, use the hub route or a notification carrying livestream_feed_id.
