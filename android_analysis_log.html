Method notes

Reusable lessons from this work. These are the things that would have changed the conclusion had they been missed — worth carrying into the next analysis.

Blended retention hides mix shift

A blended retention number falls automatically when acquisition scales, because a low-retaining new cohort takes a larger share of the base — even if no segment gets worse. It's Simpson's paradox in a metric people check weekly. Decompose the blend by cohort share before diagnosing a "regression"; back out the established-user rate and check whether it actually moved. Track new-user and existing-user retention as separate series, or the blended one will keep generating false alarms as you grow.

Compare releases at matched age, not on the same calendar window

A fresh release looks artificially bad early (upgrade-path bugs, migration crashes, early-adopter skew) — not artificially good, which is the intuition most people start with. Comparing a 2-day-old release against a mature one on the same calendar window is the hardest possible test for the new build. The fair comparison is each release against the others at the same age. I initially got this backwards and had to correct it.

ANRs are not crashes, and don't appear in crash-free rate

On Android 11+, ANRs arrive via ApplicationExitInfo — reported retroactively on the next launch, flagged handled: 1, mechanism AppExitInfo. They never mark the live session crashed, so they do not feed crash-free sessions/users. ANR rate is a separate metric on a separate chart. A large ANR win can therefore be invisible in crash-free %, and a crash-free % move must be explained by something else.

Background ANRs ≠ the Play Store vital

Google's user-perceived ANR rate — the vital that governs store visibility against the 0.47% bad-behavior threshold — counts only ANRs where the app failed to respond to an input event, i.e. foreground. A fix that eliminates background ANRs (e.g. blocking work during a push-triggered cold start) can be a real product win and still barely move the Play Console number. Check which bucket your win landed in before reporting it.

Client-side error sampling distorts counts, not ratios

If the crash SDK samples errors (e.g. sampleRate = 0.01), every absolute count is a sample and the true volume is ~100×. Cross-release ratios remain valid as long as the sample rate is unchanged across the builds being compared — so check the sampling config's git history before trusting a release comparison. Also check whether any beforeSend drop rules landed in the window: a release can look better simply because it stopped reporting.

Nested JSON properties don't resolve in the analytics query API

Where an event property is stored as a JSON object, dot-notation paths silently return undefined rather than erroring — a filter written that way matches nothing and the chart comes back empty, which reads as "no data" instead of "bad query". The working equivalent is a custom property that extracts the nested key. Always validate a new query by reproducing a known-good saved total before trusting anything built on it.

Pick the denominator deliberately

"Users who emitted at least one error event" is not "active users". Rates built on the former are comparison proxies, not true crash-free/ANR-free rates, and shouldn't be quoted as percentages of the userbase. Likewise, install-cohort retention and signup-cohort retention are different populations that produce very different numbers — both correct, neither interchangeable. State which one a figure refers to, every time.
