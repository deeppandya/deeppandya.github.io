First, how the surface is shaped

Three ways an endpoint reaches a caller

Confirmed in the repo: Whop has one internal GraphQL API, and the public REST API is produced two ways. A "new endpoint" means picking one of these lanes. The raffle journey leans on all three — and a few steps don't need to be public at all.

LANE 1

Native /api/v1

Hand-written Rails controllers with dated versioning (Api-Version-Date). The newest, fastest path.

/products · /plans · /accounts · /users · /checkout_configurations

LANE 2

GraphQL → REST proxy

~59 GraphQL resources are auto-translated to REST by api_definition.rb. Add a resource/action, get a path for free.

/companies · /files · /affiliates · /account_links · /webhooks · /shipments

LANE 3

Internal core GraphQL

whop.com-only mutations, never published. Perfect for onboarding flows that don't need a developer contract.

createCompanyFromPrompt · AI "design my page" · classification setters

// the public @whop/sdk is generated from the OpenAPI that lanes 1 & 2 produce — 65 resources, no raffle

The journey, call by call

Seven steps, in order

01

whop.com → sign up

Create your seller account

Self-serve sellers authenticate with OAuth and skip this. The platform path — building a raffle on behalf of a seller — provisions an account directly.

accounts.create POST /api/v1/accounts have · native 
POST /api/v1/accounts
{
  "email": "jack@example.com"
}
→ { "id": "biz_8s2Kd…" }
# native /api/v1 — not in the proxy. Self-serve users OAuth instead.
02

start a business

Spin up the raffle business

One call creates the company. The catch is the business classification: sweepstakes_raffles exists, but it is flagged restricted — there's no clean public field to set it, and it changes what the account is allowed to do.

companies.create POST /companies have · proxy 
POST /companies
{
  "title": "Jack's Raffles",
  "email": "jack@example.com"
}
→ { "id": "biz_8s2Kd…" }

Compliance gate (blocker, not just an endpoint): the matching classification sweepstakes_raffles is FINANCING_RESTRICTED + NON_DISCOVERABLE — hidden from Discover, no buy-now-pay-later — and raffles are legally regulated. It can't be set through public REST today, and legal must sign off before launch.

03

pick the prize

Choose & photograph the item

The picker (Pokémon cards / e-bike / Robux / socks / Labubus) just pre-fills a title. Upload a photo, then create the storefront — an access pass / "product". For physical prizes, turn on shipping-address collection right here.

files.create → S3 POST /files + PUT have · proxy 
# 1 · ask for a presigned URL
POST /files
{ "filename": "charizard-psa10.png", "visibility": "public" }
→ { "id": "file_…", "upload_url": "https://s3…", "upload_headers": {…} }
# 2 · PUT the bytes, then poll until upload_status == "ready"
PUT {upload_url}   (body: image bytes)
products.create POST /api/v1/products have · native 
POST /api/v1/products
{
  "company_id": "biz_8s2Kd…",
  "title": "Charizard PSA 10 — Raffle",
  "headline": "1,000 tickets · winner drawn live",
  "route": "charizard-raffle",
  "collect_shipping_address": true
}
→ { "id": "prod_…" }
# false for digital prizes (Robux). collect_shipping_address is real.
coreDesignProductPage internal GraphQL · NEW build 
# NEW · the "prompt to design my product page" step has no endpoint
mutation { coreDesignProductPage(
  productId: "prod_…",
  prompt: "clean trading-card hype page, gold + black",
  referenceFileIds: ["file_…"]
) { fileIds copy { headline body } } }
# reuses the ads image+copy pipeline (Gemini/Flux + Claude) already wired
04

price the entry

Set ticket price & supply

A "ticket" is approximated by a one-time plan with limited stock. Price and inventory are real and shipped — but a plan models one fixed-price entry, not odds, per-user caps, or "buy 10 tickets."

plans.create POST /api/v1/plans have · native 
POST /api/v1/plans
{
  "product_id": "prod_…",
  "plan_type": "one_time",
  "initial_price": "5",        // ← ticket price
  "currency": "usd",
  "stock": 1000,              // ← total tickets
  "image": { "id": "file_…" }
}
→ { "id": "plan_…", "purchase_url": "…" }
raffles.tickets.create POST /raffles/{id}/tickets · NEW build 
# NEW · "buy N tickets" in one purchase — plans only do 1 fixed entry
POST /raffles/raf_…/tickets
{ "quantity": 10, "max_per_user": 50 }
→ { "entries": 10, "payment_id": "pay_…", "odds": "10 / 1000" }
# plans have allow_multiple_quantity, but no per-user cap or odds
05

set the deadline · the core gap

Choose when the raffle closes

This is where existing primitives run out. A plan's expiration_days is a membership expiry after purchase — not a sales cutoff. There is no "entries close at" concept anywhere, and nothing auto-stops sales or triggers a draw.

plans.update (stopgap) PATCH /api/v1/plans/{id} have · but fake 
# works, but Whop will NOT act on it — you'd run your own scheduler
PATCH /api/v1/plans/plan_…
{ "metadata": { "raffle_ends_at": "2026-07-01T00:00:00Z" } }
→ stored as a string · no enforcement, no auto-close
raffles.create POST /raffles · NEW build 
# NEW · the raffle primitive — collapses steps 3–5, owns the deadline
POST /raffles
{
  "company_id": "biz_…", "image_id": "file_…",
  "ticket_price": "5", "currency": "usd",
  "total_tickets": 1000, "max_tickets_per_user": 50,
  "ends_at": "2026-07-01T00:00:00Z",
  "winner_count": 1, "draw_mode": "auto_on_close"
}
→ { "id": "raf_…", "checkout_url": "…", "status": "open" }
# a worker closes sales + draws at ends_at — the missing backbone
06

share the raffle

Send the link & recruit partners

Sharing a checkout link and paying affiliates are both shipped and public. The raw link is free — it's just the product route from step 03.

checkoutConfigurations.create POST /api/v1/checkout_configurations have · native 
POST /api/v1/checkout_configurations
{ "plan_id": "plan_…", "redirect_url": "https://jacks-raffles.com/thanks" }
→ { "purchase_url": "https://whop.com/c/…" }
# a ready-to-share ticket-checkout link
affiliates.create POST /affiliates have · proxy 
POST /affiliates
{ "username": "hype_partner", "payout_type": "percentage",
  "fee": "10", "access_pass_ids": ["prod_…"] }
→ share link: whop.com/charizard-raffle?a=hype_partner

Web link = no API call. whop.com/charizard-raffle is just the product route. Geo-targeted promo (Meta / TikTok) reuses the existing ads stack if the seller wants paid reach.

07

after the drop · draw & deliver

Pick a winner, get paid, ship the prize

The deadline hits. Three of these are shipped (payouts, entry tracking, shipping); the two that make it a raffle — a provably-fair draw and entry transparency — don't exist yet.

raffles.draw POST /raffles/{id}/draw · NEW build 
# NEW · pick winner(s), provably fair — RNG seed + audit record
POST /raffles/raf_…/draw
{ "winner_count": 1 }
→ { "winners": [{ "user_id": "user_…" }],
     "proof": { "seed": "beacon:#4521…", "hash": "sha256:…" } }
raffles.entries.list GET /raffles/{id}/entries · NEW build 
# NEW · odds / entry transparency before and after the draw
GET /raffles/raf_…/entries
→ { "total_entries": 1000,
     "entrants": [{ "user_id": "user_…", "tickets": 10 }] }
accountLinks.create POST /account_links have · proxy 
# KYC + bank so ticket revenue can land
POST /account_links
{ "company_id": "biz_…", "use_case": "account_onboarding",
  "refresh_url": "…/refresh", "return_url": "…/done" }
→ { "url": "https://…onboarding…" }
webhooks.create POST /webhooks have · proxy 
# each payment.succeeded = a new ticket / entry
POST /webhooks
{ "url": "https://jacks-raffles.com/webhooks",
  "events": ["payment.succeeded"] }
shipments.create POST /shipments have · proxy 
# ship the prize + attach tracking (EasyPost-backed)
POST /shipments
{ "company_id": "biz_…", "payment_id": "pay_winner_…",
  "tracking_code": "1Z999AA10123456784" }

The draw is the heart of it. A raffle people trust needs verifiable randomness — a signed external beacon (random.org / drand) or on-chain VRF — plus an immutable audit record. That's the one genuinely new integration, not just a new endpoint.

The build list

New endpoints we need

One primitive carries the journey; four actions hang off it. The AI page-design call is an optional accelerator. Everything else is reuse.

B1 · core

Raffle primitive

POST /raffles

The first-class object. Owns ticket price, supply, per-user caps, ends_at, winner count & draw mode. Collapses today's product + plan + metadata hack.

native /api/v1+ close workersize: L
B2 · core

Ticket purchase

POST /raffles/{id}/tickets

Buy N entries in one payment with a per-user cap and live odds. Plans can't model multi-entry odds.

reuses paymentssize: M
B3 · core

Provably-fair draw

POST /raffles/{id}/draw

Pick winner(s) from a verifiable random seed and write an immutable audit record. No drawing service exists today.

+ RNG beaconasyncsize: M
B4 · core

Entry transparency

GET /raffles/{id}/entries

Entrants, ticket counts & current odds — what makes a raffle feel legitimate to buyers.

read-onlysize: S
B5 · optional

AI page design

coreDesignProductPage

The "prompt to design my product page" step. Wraps the ads image + copy pipeline that already exists. Internal GraphQL.

internalreuses ads AIsize: S

Open questions to settle

  • Is a raffle a first-class /raffles resource, or a thin façade over product + plan? (Recommend first-class — the deadline + draw have no home otherwise.)
  • Randomness source — signed random.org / drand beacon (cheap, off-chain) vs on-chain VRF (maximal trust, more ops)?
  • Compliance: who owns the legal review for sweepstakes_raffles, and which jurisdictions ship in v1?
  • Do third-party developers get /raffles in public REST, or is it whop.com-only at launch?
