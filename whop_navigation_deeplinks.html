Android testing (develop build)

Build: com.whop.whopapp.develop  ·  signed in as @deeppandya91.

Browser tap. Android verifies these paths (profiles, support, notifications, discover, and store sections), so a tapped link opens the app. App-Links paths: /@, /u/, /account, /support, /notifications, /discover, /*/reviews, /*/products, /*/app-store. On the develop build, verification isn't automatic — enable it once in Settings → Apps → Whop (develop) → Open by default → Open supported links → add whop.com.

adb (always works, bypasses link verification). Pushes the URL as the deeplink extra:

adb shell "am start -n com.whop.whopapp.develop/com.whop.whopapp.MainActivity \
  --es deeplink '<URL>'"

Android-specific behavior notes

  • /@username and /u/username both resolve to the same native profile, ahead of bot-route parsing.
  • /discover and /notifications resolve to their top-level tabs before bot-route parsing, so they aren't mistaken for a company route.
  • /discover/{whop-route} opens that company's store page (a discover store link); bare /discover opens the Discover tab.
  • Store sections open the matching surface: products → Products tab, app-store → Apps tab, reviews → reviews sheet.
  • Settings sections open their native screen; balance / billing-history have no native screen and open in an in-app web view. The membership/invite id in the URL is parsed but Android opens the list (no per-item highlight).
  • Single-segment store pages (e.g. whop.com/deep-ai-7c1d) are not auto-verified (Android manifest can't claim them without a catch-all) — they open via adb / in-app links only.
